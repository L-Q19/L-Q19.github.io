openclaw调教日记

发表于 2026-06-20 更新于 2026-06-21 分类于 AI学习
本文字数： 2.6k 阅读时长 ≈ 6 分钟

希望虾没事

day1

做好了部署准备:git、nodejs、ubuntu24安装完毕

通过宿主机访问WMware虚拟机的ubuntu网关成功安装openclaw

并连接大模型deepseek4.0且接入飞书进行使用（deepseek充值3人民币）

进度可谓非常顺利

day2

准备制作公交出门自动提示功能

在此基础上了解到了如下内容
1.openclaw可以使用python环境下的browser-use库来进行直接操作浏览器
2.可以通过对浏览器的操作做到对实时公交进行截图然后通过飞书发送到我的手机上，交给他我的到达站点预计时间由此可以得到合理出门时间。
3.通过broser操作浏览器需要对浏览器进行配置，但我们可以通过playwright中的chromium浏览器来实现直接接入（同使用chrome内核）

在今天我们成功下载了playwright以及他所携带的chromium
但最后一个144mb的包没有下载好，就是这个（headless-shell）但目前看来并没有什么影响

day3

了解到想要进行浏览器操作可以使用更简便的方法，在chrome144版本更新后内置的MCP协议更强大了，openclaw2026.3.13版本他更是直接拥有browser功能
我们直接打开chromium的测试远程连接端口

在这里 chrome://inspect/#remote-debugging

然后我们直接在配置文件openclaw.json修改就可以了

然后重启 OpenClaw

查看状态，并且同步激活浏览器，这是浏览器会弹窗请求运行控制

简直完美直接就是一个配置成功

后来我们依旧成功，期间进行了browser的配置以及playwright+chromium有所差异的配对设置
并且测试了他启动浏览器和测试页面，并手动使用了截图功能。均成功。

本来想直接在ubuntu部署，但是得知VMware不能直接使用GPU来进行计算故作罢决定在宿主机部署
23点56分测试飞书传截图已经成功了（续费deepseek3元）

day4

今天进行了本地ollama的配置，续费了机场（消费30元），成功部署ollama和qwen3.5：4b，成功配置了宿主机和虚拟机的通信。
但是接入openclaw后效果不尽人意

整理今天的操作

1.下载ollama（使用科学上网完成）
将ollama的主程序和模型存放位置进行了修改。并配置了两个环境变量【a.主程序在用户配置 b.模型在系统配置到.model的文件夹】

2.打通虚拟机和宿主机的连接
进行了入网协议的新增，开放了openclaw对ollama端口的访问（在系统环境变量写入了ollama的端口）

3.配置openclaw接入本地qwen3.5：4b 直接使用他的引导程序.

是openclaw的交互式引导配置，通过引导,完成 Gateway、模型、工作区、渠道等所有核心配置。

顺带一提，这个是斩杀ollama的命令

然后现在发现问题了openclaw接入后只能回复我两个字（准确来说应该是NO或者不行）
这个在调整了几次上下文设置之后还是存在问题，只能明天看能不能再下一个模型或者看他还有没有其他的配置来解决这个问题了。

day5

查阅ollama官方文档结合测试内容来看

1.ollama官方文档有内容：

Ollama 的 API 响应包含可用于衡量性能和模型使用情况的指标
total_duration：生成响应所花费的总时间
load_duration：加载模型所花费的时间
prompt_eval_count：处理的输入 token 数量
prompt_eval_duration：评估提示词所花费的时间
eval_count：处理的输出 token 数量
eval_duration：生成输出 token 所花费的时间
所有时间值均以纳秒为单位测量。

2.在虚拟机使用直接访问ollama的形式

得到以下内容

提取以下内容
“content”:”” 发现回复为空
done_reason”:”length” 意为模型生成的内容因达到最大长度限制而被截断。

由此我们可以得出结论，是qwen3.5的思考内容太过冗长，达到了token最大限制，没有返回给openclaw答案。
经过验证也发现，直接使用ollama来与qwen3.5：4b对话也常常在思考时中断没有回复给我们结果。

目前不清楚是ollama配置问题还是openclaw配置问题

得到以下两个方案，关掉千问的思考模式，但在初步尝试修改openclaw的参数，没有起到效果。
现有两个想法，一是qwen3.5：4b这个模型本身不支持关闭思考模式，它原本就是思考模型。二是在openclaw进行设置不能起到作用。

后又使用

在访问中直接添加no_think参数依然会失败。我们或许只能选择放弃qwen3.5这个模型了?

day6

下载了qwen3.5：9b和4b的版本，通过一些列的调试
（添加think_no等参数或者加大maxTokens的量，都没办法解决content=空的问题）
结果均失败，有两个方向的怀疑，1.ollama的封装问题 2.qwen3.5模型的设置问题

并且在尝试中也试着添加参数然后重新构建一个不思考的qwen，但结果依旧失败。尝试步骤如下

到第三步就识别不到 think 这个参数了，计划破产说是

只能用折中的办法，在每次和他聊天的时候都要使用think off 的指令，才能让他短暂的回话。
但这个程度还远远不够，他只有deepseek4pro一半的能力。
接入deepseek，让他从本地发文件和浏览器用了不到二十分钟。
但qwen3.5：9b还是太局限了。抑或是我的显卡太弱小了，根本无法带的动这样的工具，
让他给我发文字需要重新开一个会话，让他给我发txt文件他也只能把简短的内容发给我，文件始终成功不了
而且会话长了他就会忘掉上下文或者每句话都只说一半，我怀疑是显存爆了。期间显卡占用率甚至达到了惊人的百分百
如果只是对话模型我不如直接使用免费对话模型不是吗？
难道本地部署大模型是骗局吗？付费cloud模型才是真相啊…
（或者是ollama限制了模型的能力？其实他应该还是挺强的？换个模型再试试吧？）

顺带一提：qwen3.5关闭思考的参数应该是 “enable_thinking”:false
但存在一个问题，他的think部分依旧有内容，只是做了调度让他不至于回复（content）一个空值
